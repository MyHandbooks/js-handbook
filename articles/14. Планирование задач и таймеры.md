Планирование выполнения (scheduling) — это возможность отложить запуск функции на определенное время или настроить её регулярный циклический вызов. В JavaScript, работающем в однопоточной среде браузера, планирование реализуется через встроенные функции веб-окружения: `setTimeout` и `setInterval`.

---

## 1. Глубокое понимание концепции (Core Concept)

### 1. Метод `setTimeout`
Позволяет запустить переданную функцию один раз после того, как пройдет указанная задержка.

Системный синтаксис:
<code>let timerId = setTimeout(func|code, [delay], [arg1], [arg2], ...)</code>

* `func|code` — функция, которую нужно выполнить. Исторически туда можно передать строку с кодом (например, `"alert('Привет!')"`), но это считается плохой практикой из-за проблем с безопасностью и производительностью. Всегда следует передавать ссылку на функцию.
* `delay` — задержка перед запуском в миллисекундах (<code>1000 мс = 1 с</code>). Значение по умолчанию — `0`.
* `arg1, arg2...` — необязательные аргументы, которые браузер автоматически передаст в вашу функцию при вызове.

Функция `setTimeout` мгновенно возвращает числовой идентификатор таймера (`timerId`). Если запуск функции больше не нужен, запланированную задачу можно отменить, вызвав метод `clearTimeout(timerId)`.

---

### 2. Метод `setInterval`
Позволяет запускать функцию регулярно через строго определенные равные интервалы времени.

Системный синтаксис:
<code>let intervalId = setInterval(func, delay, ...args)</code>

Для отмены бесконечного цикла вызовов используется метод `clearInterval(intervalId)`.

---

### 3. Сравнение: Рекурсивный `setTimeout` против `setInterval`

Для регулярных запусков можно использовать как `setInterval`, так и рекурсивный вызов `setTimeout` (когда функция в конце своей работы планирует свой следующий запуск). Между этими подходами есть фундаментальная разница:

#### С `setInterval`:
Интервал отсчитывается от начала выполнения функции. Если сама функция выполняется долго (например, делает сложный расчет или сетевой запрос), то время работы функции поглощает интервал ожидания.
* Если функция выполняется дольше, чем заданный интервал, то вызовы начнут происходить без пауз один за другим, перегружая поток выполнения.

#### С рекурсивным `setTimeout`:
Следующий запуск планируется только тогда, когда текущий вызов полностью завершился. Это гарантирует строго фиксированную паузу между окончанием работы функции и началом её следующего запуска, независимо от того, сколько времени заняло выполнение кода.

---

### 4. Нулевая задержка: `setTimeout(f, 0)`
Передача нулевой задержки (или вызов `setTimeout(f)`) планирует запуск функции при первой возможности, но **не мгновенно**.
* Движок JavaScript сначала завершит выполнение всего текущего синхронного кода на странице.
* Только после того, как стек вызовов полностью освободится, движок возьмет задачу из очереди макрозадач и выполнит её.
* **Минимальная задержка:** Согласно стандартам HTML5, после 5 последовательных вложенных вызовов таймера браузер автоматически увеличивает минимальную задержку до `4 мс`.

---

### 5. Память и сборка мусора
Когда вы передаете функцию в `setTimeout` или `setInterval`, браузер создает внутреннюю ссылку на эту функцию, чтобы запустить её в будущем. Это предотвращает удаление функции и всех переменных из её замыкания сборщиком мусора.
* Если `setInterval` не очистить с помощью `clearInterval`, функция и связанные с ней данные будут оставаться в памяти бесконечно, даже если объект, вызвавший её, давно удален со страницы. Это частая причина утечек памяти.

## 2. Ментальная модель

Разница в распределении времени между вызовами при тяжелых вычислениях:

```text
setInterval(heavyFunc, 1000):
[ Ожидание 1с ] -> [heavyFunc (длится 1.5с)]
                   |
                   +--- [ Очередной интервал 1с подошел! Функция уже должна была запуститься! ]
                        (Вызовы наползают друг на друга, интервалы сбиваются).

Рекурсивный setTimeout(heavyFunc, 1000):
[ Ожидание 1с ] -> [heavyFunc (длится 1.5с)] -> [ Ожидание 1с гарантированно ] -> [ heavyFunc ]
(Задержка стабильна и не зависит от скорости работы функции).
```

## 3. Практические примеры кода

### Плохой пример: использование setInterval для тяжелых задач и утечка памяти

```javascript
// Передача строки кода вместо функции (плохая практика)
setTimeout("console.log('Не делайте так')", 1000);

// Запуск тяжелой задачи через setInterval
const badTimerId = setInterval(() => {
  // Тяжелые вычисления, которые могут занять более 500 мс
  for (let i = 0; i < 999999999; i++) {} 
  console.log("Цикл отработал.");
}, 500);

// Забыли очистить badTimerId при уходе со страницы — функция будет вечно крутиться в памяти
```

### Хороший пример: рекурсивный setTimeout и правильная очистка таймеров

```javascript
// 1. Использование рекурсивного setTimeout для гарантированного интервала
let userRequestCount = 0;

function fetchUserData() {
  userRequestCount++;
  console.log(`Запрос данных пользователя... Попытка №${userRequestCount}`);

  // Имитируем сетевой запрос разной длительности
  const requestDuration = Math.random() * 2000;

  setTimeout(() => {
    console.log(`Данные получены. Запрос длился ${Math.round(requestDuration)} мс`);

    // Планируем следующий запрос строго через 3 секунды ПОСЛЕ получения ответа
    if (userRequestCount < 5) {
      setTimeout(fetchUserData, 3000);
    } else {
      console.log("Лимит запросов исчерпан. Планирование завершено.");
    }
  }, requestDuration);
}

// Запускаем первую итерацию
fetchUserData();

// 2. Правильная очистка интервала при наступлении условия
let secondsPassed = 0;

const clockId = setInterval(() => {
  secondsPassed++;
  console.log(`Прошло секунд: ${secondsPassed}`);

  if (secondsPassed >= 3) {
    // Обязательно останавливаем таймер, чтобы освободить ресурсы и не допустить утечки памяти
    clearInterval(clockId);
    console.log("Таймер успешно остановлен.");
  }
}, 1000);
```

## 4. Вопросы для собеседования

### Вопрос 1: Почему рекурсивный `setTimeout` считается более безопасным для периодических задач, чем `setInterval`?
**Ответ:** `setInterval` не учитывает время выполнения самой функции-обработчика. Он планирует запуски строго по часам. Если выполнение функции занимает больше времени, чем установленный интервал, вызовы начнут накладываться друг на друга без пауз, что может привести к зависанию интерфейса. 

Рекурсивный `setTimeout` планирует следующий запуск только тогда, когда текущий вызов полностью завершился, гарантируя стабильную паузу между вызовами.

### Вопрос 2: Выполнится ли функция в `setTimeout(f, 0)` мгновенно? Какова реальная задержка?
**Ответ:** Нет, она не выполнится мгновенно. Она помещается в очередь макрозадач (Macro-task queue) и будет выполнена только после того, как текущий синхронный код в стеке вызовов полностью завершится. 

Даже при указании `0 мс` реальная задержка зависит от загруженности процессора другими задачами. Кроме того, по спецификации HTML5, после 5 вложенных вызовов таймера браузер принудительно увеличивает минимальную задержку до `4 мс` (а во вкладках, находящихся в фоновом режиме, задержка может увеличиваться до `1000 мс` для экономии батареи).

### Вопрос 3: Что произойдет с запланированным `setTimeout` таймером, если основной поток JavaScript будет заблокирован синхронной операцией на 5 секунд?
**Ответ:** Таймер не сможет выполниться вовремя. Среда выполнения JavaScript однопоточна. Когда таймер обратного отсчета истекает в фоновом режиме (Web APIs), его колбэк отправляется в очередь макрозадач. Он будет ожидать там до тех пор, пока тяжелая синхронная задача в стеке вызовов не завершится. Код выполнится с опозданием (примерно через 5 секунд плюс исходная задержка).

### Вопрос 4: Как таймеры могут вызывать утечки памяти?
**Ответ:** Колбэк-функция, переданная в `setTimeout` или `setInterval`, остается в памяти браузера до тех пор, пока таймер не сработает (для `setTimeout`) или не будет явно очищен (для `setInterval`). Если эта функция содержит замыкание (ссылается на внешние объекты, элементы DOM или переменные), эти данные также не могут быть удалены сборщиком мусора. Неочищенный `setInterval` будет вечно удерживать связанные с ним объекты в памяти.
