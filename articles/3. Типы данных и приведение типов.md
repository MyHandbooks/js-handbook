В JavaScript все типы данных строго разделены на две категории: **Примитивы (Primitives)** и **Сложные типы (Объекты)**. Они различаются принципом хранения в оперативной памяти компьютера и особенностями копирования.

---

## 1. Глубокое понимание концепции (Core Concept)

### Примитивы против Объектов
* **Примитивы** хранятся в быстрой памяти (стеке) непосредственно как значения. Они неизменяемы (immutable). При копировании примитива создается его независимая копия в новой ячейке памяти.
* **Объекты** хранятся в динамической памяти (куче), а переменная в стеке хранит лишь ссылку (адрес ячейки в куче) на этот объект. При копировании объекта копируется только эта ссылка, а сам объект в памяти остается в единственном экземпляре.

---

### Восемь типов данных в JavaScript

Движок JavaScript динамически определяет тип переменной в момент присвоения значения (динамическая типизация).

1. **Number:** Любые числа (целые и дробные), а также три специальных числовых значения:
   * `Infinity` (математическая бесконечность, например `1 / 0`).
   * `-Infinity` (отрицательная бесконечность).
   * `NaN` (Not-a-Number — ошибка математического вычисления, например `"строка" * 5`).
2. **BigInt:** Для работы с целыми числами любой длины, выходящими за безопасные пределы типа Number. Создаются добавлением буквы `n` в конец числа: `9007199254740991n`.
3. **String:** Строки. В JS нет отдельного символьного типа (char), символ — это просто строка из одного элемента.
4. **Boolean:** Логический тип, имеющий только два значения: `true` (истина) и `false` (ложь).
5. **null:** Специальное значение, обозначающее «пусто», «ничего» или «значение неизвестно». Это намеренное указание разработчика на отсутствие значения.
6. **undefined:** Означает, что переменная объявлена в памяти, но значение ей ещё не присвоено (дефолтное состояние памяти).
7. **Symbol:** Уникальные и неизменяемые идентификаторы, используемые для создания скрытых свойств объектов.
8. **Object:** Сложная структура данных, объединяющая коллекции свойств (сюда же относятся массивы `[]`, функции `function()`, даты `new Date()`).

### Оператор `typeof` и его аномалии
Оператор `typeof` возвращает строку с именем типа данных проверяемого значения. В его работе есть две важные исторические аномалии:
1. <code>typeof null === "object"</code> — это общепризнанная ошибка в исходном коде движка JS. В ранней реализации JavaScript значения хранились в виде 32-битных слов, где тип определялся по первому маркеру (тегу типа). Для объектов тег был равен `000`, а значение `null` представляло собой нулевой указатель (все биты равны 0), из-за чего движок ошибочно распознал его как объект. Ошибку оставили ради обратной совместимости.
2. `typeof function() {}` возвращает `"function"`. Спецификация выделяет функции в отдельную категорию для удобства, хотя на самом деле функции под капотом являются объектами.

### Приведение типов (Type Coercion)

Приведение типов бывает **явным** (разработчик сам вызывает функции преобразования, например `Number(val)`) и **неявным** (движок JS выполняет его автоматически из-за операторов сравнения, сложения, вычитания или проверок).

#### Логическое преобразование (ToBoolean)
Происходит внутри условий `if` и логических операторов (`||`, `&&`, `!`). Только эти **8 значений** приводятся к `false` (их называют **Falsy-значениями**):
1. `false`
2. `0`
3. `-0`
4. `0n` (BigInt ноль)
5. `""` (пустая строка)
6. `null`
7. `undefined`
8. `NaN`

*Все остальные значения (включая пустой массив `[]`, пустой объект `{}` и строку `"0"`) всегда приводятся к `true` (их называют **Truthy**).*

#### Числовое преобразование (ToNumber)
Происходит при математических операциях и принудительном приведении через унарный плюс `+val`.
* **Правила приведения примитивов:** `undefined` приводится к `NaN`, `null` — к `0`, `true` — к `1`, `false` — к `0`, пустая строка — к `0`.
* **Приведение объектов к числу:** Объект сначала преобразуется в строковый примитив (через скрытый вызов метода `toString()` или `valueOf()`), а затем полученная строка приводится к числу. Например, пустой массив `[]` сначала преобразуется в пустую строку `""`, а `Number("")` возвращает `0`.

---

### Сравнение: Строгое против Нестрогого

#### Строгое равенство (<code>===</code>)
Сравнивает значения **без приведения типов**. Если типы операндов различаются, сравнение сразу возвращает `false`.
* **Исключение:** Значение `NaN` не равно ничему в мире, включая себя. Поэтому <code>NaN === NaN</code> всегда возвращает `false`.

#### Нестрогое равенство (<code>==</code>)
Если типы различаются, движок пытается привести операнды к числам:
1. **Сравнение null и undefined:** Выражение <code>null == undefined</code> всегда возвращает `true` (это жестко зафиксировано в спецификации). Больше ни с какими другими значениями при нестрогом сравнении они не равны (например, <code>null == 0</code> вернет `false`).
2. **Boolean с другими типами:** Boolean-значение приводится к числу (`true` -> `1`, `false` -> `0`).
3. **String и Number:** Строка приводится к числу.
4. **Объект и Примитив:** Объект преобразуется в примитив через `toString()` / `valueOf()`, после чего выполняется сравнение.

## 2. Ментальная модель

### Схема хранения данных в оперативной памяти:
```text
  [ СТЕК (Быстрая память) ]                [ КУЧА (Динамическая память) ]
  -------------------------                -----------------------------
  let age = 20; (Примитив)                 
  let name = "Алекс"; (Примитив)           
  let user = [ссылка на адрес #01] ------> #01: { firstName: "Иван", age: 30 } (Объект)
```

### Схема приведения типов при сравнении <code>[] == ![]</code>:
```text
Шаг 1: Правая часть ![] приводится к Boolean. Так как [] это true, то ![] -> false.
       Выражение: [] == false

Шаг 2: По правилам, Boolean (false) приводится к числу 0.
       Выражение: [] == 0

Шаг 3: Объект [] сравнивается с числом 0. Объект приводится к примитиву: [].toString() -> "".
       Выражение: "" == 0

Шаг 4: Строка "" сравнивается с числом 0. Строка "" приводится к числу 0.
       Выражение: 0 == 0  ===> Результат: true
```

## 3. Практические примеры кода

### Плохой пример: неосознанное использование нестрогого сравнения

Использование `==` часто маскирует ошибки типов, приводя к скрытым логическим багам, а прямое сравнение с `NaN` никогда не сработает.

```javascript
const userAge = "0";

// Нестрогое сравнение приводит "0" к числу 0, а 0 к false
if (userAge == false) {
  // Код выполнится, хотя в переменной лежит строка "0" (truthy-значение)
  console.log("Возраст не указан!"); 
}

const calculation = "привет" * 5; // NaN

// Попытка проверить на NaN прямым сравнением
if (calculation === NaN) {
  // Этот блок никогда не выполнится, так как NaN не равен самому себе
  console.log("Ошибка вычислений"); 
}
```

### Хороший пример: строгое сравнение, проверка на NaN и безопасные проверки

```javascript
const userAge = "0";

// Всегда используем строгое сравнение ===
if (userAge === "0") {
  console.log("Возраст равен строковому нулю.");
}

const calculation = "привет" * 5; // NaN

// Использование специального статического метода для проверки на NaN
if (Number.isNaN(calculation)) {
  console.log("Вычисление завершилось ошибкой NaN.");
}

// Альтернативный способ проверки на NaN без методов (NaN - единственный, кто не равен себе)
if (calculation !== calculation) {
  console.log("Это значение гарантированно является NaN.");
}

// Безопасное логическое приведение через двойное отрицание !!
const hasItems = !![].length; // Превратит 0 в false, а числа больше 0 в true
console.log(hasItems); // false
```

## 4. Вопросы для собеседования

### Вопрос 1: В чем разница между `null` и `undefined` в JavaScript?
**Ответ:** 
* `undefined` — это дефолтное состояние памяти. Оно означает, что переменная была создана, но ей еще не присвоили никакого значения.
* `null` — это намеренное указание разработчика на отсутствие значения. Он используется, когда мы хотим показать, что переменная пуста или значение пока неизвестно.

### Вопрос 2: Почему <code>typeof null</code> возвращает `"object"`? Это баг или фича?
**Ответ:** Это общепризнанный исторический баг в исходном коде языка. В первой версии JavaScript значения кодировались в памяти в виде 32-битных слов, где первые несколько бит (тег) определяли тип данных. Для объектов тег был равен `000`. Поскольку значение `null` представляло собой пустой указатель (все биты равны 0), его первые биты также распознавались как `000`, из-за чего оператор `typeof` ошибочно посчитал его объектом. Ошибку оставили неисправленной для сохранения совместимости со старыми сайтами.

### Вопрос 3: Как работает нестрогое сравнение <code>==</code> под капотом, если типы операндов различаются?
**Ответ:** При нестрогом сравнении <code>==</code> движок JavaScript пытается привести типы операндов к числам по следующим правилам:
1. Если один из операндов Boolean, он приводится к числу (`true` -> `1`, `false` -> `0`).
2. Если сравниваются строка и число, строка приводится к числу.
3. Если сравниваются объект и примитив, объект приводится к примитиву (через `toString()` / `valueOf()`), а затем сравнение повторяется.
4. Единственное исключение: <code>null == undefined</code> сразу возвращает `true` без приведения типов к числам.

### Вопрос 4: Как правильно проверить, что значение в переменной является `NaN`?
**Ответ:** Сравнение <code>x === NaN</code> не сработает, так как `NaN` уникален и не равен даже самому себе. Правильные способы проверки:
1. Использовать метод `Number.isNaN(x)`. Он проверяет, что тип значения — действительно Number, и оно равно `NaN` (в отличие от старого `isNaN()`, который приводил аргументы к числу).
2. Сравнить переменную с самой собой: <code>x !== x</code>. Это выражение вернет `true` только в том случае, если в `x` действительно лежит `NaN`.
