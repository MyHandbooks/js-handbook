Конструкция `async/await` — это современный синтаксический сахар над промисами, который позволяет писать асинхронный код в синхронном стиле. Он делает код более линейным, избавляет от необходимости писать цепочки методов `.then()` и `.catch()` и упрощает отладку.

---

## 1. Глубокое понимание концепции (Core Concept)

### 1. Ключевое слово `async`
Ключевое слово `async` ставится перед объявлением функции. Оно гарантирует две вещи:
1. Функция **всегда возвращает промис**.
2. Если функция возвращает обычное значение (число, строку, объект), движок JavaScript автоматически оборачивает это значение в успешно завершившийся промис (эквивалентно вызову <code>Promise.resolve(value)</code>).

---

### 2. Ключевое слово `await`
Ключевое слово `await` заставляет интерпретатор JavaScript приостановить выполнение функции на текущей строке до тех пор, пока промис справа от `await` не выполнится. 

После выполнения промиса `await` возвращает его результат, и выполнение кода внутри функции продолжается:
<code>let result = await promise</code>

#### Особенности работы `await`:
* **Ограничение использования:** Ключевое слово `await` можно использовать **только внутри `async`-функций** (за исключением модулей верхнего уровня, где поддерживается Top-level await).
* **Неблокирующий характер:** Хотя `await` приостанавливает выполнение конкретной асинхронной функции, он **не блокирует основной поток выполнения (Call Stack) и процессор**. Пока промис ожидает разрешения, JS-движок может выполнять другие задачи: обрабатывать клики пользователей, рендерить анимацию или выполнять сторонние скрипты.

---

### 3. Обработка ошибок в асинхронном коде

В отличие от стандартных цепочек промисов, где для отлова ошибок используется метод `.catch()`, при работе с `async/await` применяется классическая синхронная конструкция `try...catch`.

#### Как работает `try...catch`:
1. Сначала выполняется код внутри блока `try { ... }`.
2. Если в процессе выполнения (включая ожидание промисов через `await`) возникает ошибка или промис отклоняется (`reject`), выполнение блока `try` прерывается.
3. Поток управления мгновенно переходит в начало блока `catch(error)`. Переменная `error` получает сгенерированный объект ошибки.

#### Объект `Error`
Когда в JavaScript происходит сбой, генерируется объект ошибки. Встроенные ошибки имеют два обязательных стандартных свойства:
* **`name`** — имя класса ошибки (например, `"ReferenceError"` при обращении к необъявленной переменной, `"TypeError"` при вызове несуществующего метода).
* **`message`** — текстовое описание деталей сбоя.
* **`stack`** — (нестандартное, но поддерживаемое всеми браузерами свойство) строка, содержащая стек вызовов, которые привели к ошибке. Используется для отладки.

#### Оператор `throw`
Позволяет генерировать ошибки вручную: <code>throw new Error("Описание ошибки")</code>. 
Технически через `throw` можно выбросить любое значение (число, строку), но хорошей практикой считается генерировать именно объекты классов, унаследованных от `Error`, чтобы сохранять совместимость с системными ошибками и записывать стек вызовов.

## 2. Ментальная модель

Сравнение структуры кода при использовании цепочки промисов и `async/await`:

```text
С цепочкой .then():
fetchUser()
  .then(user => fetchAvatar(user))
  .then(avatar => render(avatar))
  .catch(err => handleError(err)); // "Лестница" вызовов

С async/await:
try {
  const user = await fetchUser();     // Пауза выполнения до разрешения промиса
  const avatar = await fetchAvatar(user);
  render(avatar);
} catch (err) {
  handleError(err);                   // Синхронный стиль обработки ошибок
}
```

## 3. Практические примеры кода

### Плохой пример: смешивание async/await и .then/.catch

Использование старых цепочек методов `.then()` внутри современной `async`-функции усложняет код и ухудшает отслеживание ошибок.

```javascript
// Имитация API запроса
const fetchPrice = () => Promise.resolve(100);

// Плохой подход: смешивание стилей в одной функции
async function showProductPrice() {
  console.log("Загрузка...");

  // Использование then внутри async-функции лишает async/await смысла
  fetchPrice().then((price) => {
    console.log(`Цена: ${price} USD`);
  }).catch((err) => {
    console.error("Ошибка сети", err);
  });
}
```

### Хороший пример: чистый асинхронный код с try...catch и кастомной ошибкой

```javascript
const fetchProduct = (id) => {
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      if (id === 1) {
        resolve({ id: 1, title: "Ноутбук", price: 1200 });
      } else {
        // Отклоняем промис с объектом ошибки
        reject(new Error("Товар не найден в базе данных."));
      }
    }, 1000);
  });
};

// Хороший подход: линейный асинхронный код
async function displayProductDetails(productId) {
  console.log("Инициирован запрос товара...");

  try {
    // Ждем результат выполнения промиса
    const product = await fetchProduct(productId); 
    
    // Код ниже выполнится только после успешного завершения await
    console.log(`Успех! Товар: ${product.title}, Цена: ${product.price} USD`);

    // Пример ручной генерации ошибки при бизнес-логике
    if (product.price > 1000) {
      throw new Error("Превышен лимит стоимости для экспресс-доставки.");
    }

  } catch (error) {
    // Блок catch поймает как ошибку reject из промиса, так и ручной throw
    console.error("--- ОБРАБОТКА СБОЯ ---");
    console.error(`Тип ошибки: ${error.name}`);
    console.error(`Сообщение: ${error.message}`);
    // console.error(error.stack); // Для детального логирования в отладке
  }
}

// Демонстрация работы
displayProductDetails(1); // Вызовет ошибку лимита цены в блоке try
displayProductDetails(2); // Вызовет ошибку "Товар не найден" из reject промиса
```

## 4. Вопросы для собеседования

### Вопрос 1: Блокирует ли ключевое слово `await` поток выполнения браузера?
**Ответ:** Нет, `await` не блокирует основной поток выполнения. Он приостанавливает выполнение конкретной `async`-функции, в которой он вызван. В это время движок JavaScript продолжает обрабатывать другие события (клики, сетевые ответы) и выполнять синхронный код в стеке вызовов. Как только ожидаемый промис перейдет в состояние *settled*, выполнение приостановленной `async`-функции возобновится.

### Вопрос 2: Что произойдет, если использовать ключевое слово `await` внутри обычной (не `async`) функции?
**Ответ:** Это приведет к синтаксической ошибке `SyntaxError` на этапе компиляции/парсинга кода. Браузер сообщит, что `await` разрешено использовать только в асинхронных функциях. Исключением является использование `await` на верхнем уровне (Top-level await) внутри ES-модулей (файлы с тегом <code>type="module"</code>).

### Вопрос 3: Гарантирует ли объявление функции с ключевым словом `async`, что весь её код будет выполнен асинхронно?
**Ответ:** Нет. Код внутри `async`-функции выполняется абсолютно синхронно до тех пор, пока интерпретатор не встретит первое ключевое слово `await`. Только после этого выполнение функции приостанавливается, а управление возвращается в вызвавший поток, делая дальнейшую работу функции асинхронной.

### Вопрос 4: Почему рекомендуется всегда передавать экземпляр `new Error()` в оператор `throw` вместо обычной строки или числа?
**Ответ:** Передача объекта класса `Error` гарантирует, что браузер автоматически запишет текущий стек вызовов (`stack trace`) на момент генерации ошибки. Это критически важно для отладки. Если выбросить простую строку: <code>throw "Сбой"</code>, свойство `stack` будет отсутствовать, что усложнит поиск источника ошибки в больших проектах.
