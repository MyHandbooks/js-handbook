Объявить переменную в JavaScript — это значит зарезервировать именованное место в оперативной памяти компьютера для хранения данных. Исторически в языке существовал только один способ создания переменных (`var`), но стандарт ES6 (2015 год) ввел более безопасные и предсказуемые инструменты: `let` и `const`.

---

## 1. Глубокое понимание концепции (Core Concept)

### Сравнение ключевых слов по трем критериям

#### А. Область видимости (Scope)
Область видимости определяет границы кода, внутри которых переменная доступна для чтения и записи.
* **Блочная область видимости (`let`, `const`):** Переменная ограничена ближайшими фигурными скобками `{}` (блоком кода). Блоком может быть тело цикла `for`, условие `if`, функция или даже просто пустые фигурные скобки. За пределами этого блока переменная физически не существует.
* **Функциональная область видимости (`var`):** Переменная, созданная через `var`, ограничена только телом функции, в которой она объявлена. Блочные конструкции вроде `if` или `for` она полностью игнорирует и "выходит" за их пределы.

#### Б. Повторное объявление и переопределение значения
* `var` разрешает повторно объявлять переменную с тем же именем в одной области видимости, что часто приводило к случайному затиранию данных.
* `let` и `const` строго запрещают повторное объявление в той же области видимости, выбрасывая синтаксическую ошибку `SyntaxError`.
* `const` (сокращение от constant) запрещает перезаписывать само значение переменной после её инициализации. Попытка переопределения вызовет `TypeError`. При этом, если в `const` записан объект или массив, изменять свойства объекта или элементы массива разрешено, так как сама ссылка в памяти не меняется.

### 2. Поднятие (Hoisting) и Временная мертвая зона (TDZ)
При запуске кода движок JavaScript сначала сканирует его на наличие объявлений переменных и функций, мысленно перемещая их в начало их областей видимости. Этот процесс называется **поднятием (hoisting)**.

* **Поднятие `var`:** Движок находит переменную `var`, создает её в памяти и автоматически присваивает ей начальное значение `undefined`. Из-за этого к переменной можно обратиться еще до строки её объявления в коде (она вернет `undefined`).
* **Поднятие `let` и `const` (TDZ):** Эти переменные также поднимаются движком, но они **не инициализируются** значением `undefined`. Они остаются в состоянии "неинициализировано" в памяти.

Интервал времени от момента входа в область видимости до строки, где физически объявлена переменная `let` или `const`, называется **Временной мертвой зоной (Temporal Dead Zone, TDZ)**. Любое обращение к переменной в этот период вызовет ошибку `ReferenceError`.

### 3. Лексическое окружение (Lexical Environment) под капотом
Каждый запускаемый блок кода `{}` (или функция) имеет скрытый внутренний объект — **Лексическое окружение (Lexical Environment)**. Оно состоит из двух частей:
1. **Environment Record (Запись окружения):** Локальная таблица, где хранятся переменные, объявленные в этом блоке, и параметры функций.
2. **Outer Reference (Ссылка на внешнее окружение):** Указатель на родительское лексическое окружение (код, который находится "снаружи" текущего блока).

При поиске переменной движок JS сначала заглядывает в локальную таблицу `Environment Record`. Если её там нет, он по ссылке `Outer Reference` переходит во внешнее окружение и ищет там. Поиск продолжается по всей цепочке областей видимости (Scope Chain) до Глобального окружения (`Global Scope`). Если переменная не найдена и там, генерируется ошибка `ReferenceError`.

### 4. Ловушка с `var` в цикле
Классическая проблема однопоточного JavaScript — асинхронное выполнение кода внутри циклов с использованием `var`.

```javascript
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100); 
}
// Выведет в консоль: 3, 3, 3 (а не 0, 1, 2)
```
Поскольку `var` игнорирует блочные скобки цикла, переменная `i` создается в единственном экземпляре на весь скрипт. К моменту выполнения колбэков `setTimeout` (через 100 мс) цикл уже завершился, и значение переменной стало равным `3`. Все три функции ссылаются на одну и ту же переменную в памяти.

Использование `let` решает эту проблему: так как `let` имеет блочную область видимости, при каждой итерации цикла создается **новое лексическое окружение со своей независимой копией переменной `i`**.

## 2. Ментальная модель

### Схема поиска переменных по Scope Chain:
```text
[ Глобальное окружение ] <--- (Outer Reference) --- [ Окружение функции ] <--- (Outer Reference) --- [ Окружение блока if ]
 (Переменная x)                                     (Параметры, var b)                              (let c)
```

### Схема Hoisting (Поднятие) для `var` и `let`:
```text
Область видимости:
|--------------------------------------|
| (Вход в область)                     |
| var a; // Получает undefined         | <--- Обращение вернет undefined
| let b; // Не инициализировано (TDZ)  | <--- Обращение выбросит ReferenceError (TDZ)
|                                      |
| console.log(a); // undefined         |
|                                      |
| a = 5;  // Инициализация             |
| b = 10; // Инициализация (Выход из TDZ)
|--------------------------------------|
```

## 3. Практические примеры кода

### Плохой пример: утечка областей видимости из-за `var`

Использование `var` и случайный пропуск объявлений переменных приводят к засорению глобальной памяти и багам с асинхронностью.

```javascript
function processData() {
  var isValid = true;

  if (isValid) {
    // Переменная data утекает за пределы блока if
    var data = "Секретные данные"; 
  }

  // Переменная data все еще доступна, хотя блок if завершился
  console.log(data); // Вывод: "Секретные данные"

  // Случайный пропуск объявления создает глобальную переменную
  globalValue = "Я нахожусь в window"; 
}
processData();
```

### Хороший пример: изоляция переменных через `let` и `const`

```javascript
function processDataSecure() {
  const isValid = true;

  if (isValid) {
    // Переменная надежно заблокирована внутри фигурных скобок if
    const secureData = "Секретные данные";
    console.log(secureData);
  }

  // Попытка обратиться к secureData вызовет ошибку
  // console.log(secureData); // Uncaught ReferenceError: secureData is not defined

  // Безопасная работа с циклом и асинхронностью
  for (let i = 0; i < 3; i++) {
    // Для каждой итерации создается отдельная переменная i в памяти
    setTimeout(() => {
      console.log(`Итерация: ${i}`); // Выведет корректно: 0, 1, 2
    }, 100);
  }
}
processDataSecure();
```

## 4. Вопросы для собеседования

### Вопрос 1: В чем разница между `var`, `let` и `const`?
**Ответ:** Различия заключаются в трех аспектах:
1. **Область видимости:** `var` имеет функциональную область видимости, а `let` и `const` — блочную (ограничены скобками `{}`).
2. **Переопределение:** `const` запрещает перезаписывать значение переменной (но позволяет мутировать свойства объектов и элементы массивов по ссылке). `let` и `var` позволяют свободно менять значения.
3. **Поднятие (Hoisting):** `var` при поднятии инициализируется значением `undefined`. Переменные `let` и `const` тоже поднимаются, но не инициализируются, создавая Временную мертвую зону (TDZ). Обращение к ним до строки объявления вызовет ошибку `ReferenceError`.

### Вопрос 2: Что такое Temporal Dead Zone (Временная мертвая зона)?
**Ответ:** Временная мертвая зона (TDZ) — это состояние переменной, объявленной с помощью `let` или `const`, в период от момента входа в ее область видимости (например, запуск блока кода `{}`) до строки, где эта переменная фактически объявляется в коде. Попытка прочитать или записать переменную в этот период приводит к ошибке `ReferenceError`.

### Вопрос 3: Можно ли изменить свойства объекта, если он объявлен через `const`? Почему?
**Ответ:** Да, можно. Ключевое слово `const` защищает от перезаписи саму переменную (ссылку на объект в памяти). Оно запрещает выполнять операции присваивания вроде <code>myObject = newObject</code>. Однако свойства самого объекта, на который указывает эта ссылка, изменять можно, так как адрес объекта в памяти при этом не меняется.

### Вопрос 4: Как работает Scope Chain (цепочка областей видимости) под капотом лексического окружения?
**Ответ:** При поиске переменной движок заглядывает в текущее лексическое окружение (`Lexical Environment`). Если переменная не найдена в его локальной записи (`Environment Record`), движок использует ссылку на внешнее окружение (`Outer Reference`) и переходит к родителю. Этот процесс рекурсивно повторяется вверх по цепочке до глобального лексического окружения (`Global Scope`). Если переменная не обнаружена и там, выбрасывается ошибка `ReferenceError`.
