Веб-хранилища `localStorage` и `sessionStorage` предоставляют разработчикам возможность хранить данные в формате "ключ-значение" прямо на компьютере пользователя в его браузере. Это позволяет сохранять настройки интерфейса, состояние корзины покупок или данные сессии без необходимости отправлять их на сервер при каждом сетевом запросе (в отличие от Cookies).

---

## 1. Глубокое понимание концепции (Core Concept)

### 1. Правило ограничения источника (Same-Origin Policy)
Данные в браузерных хранилищах жестко привязаны к **источнику (Origin)**. Источник определяется комбинацией трех параметров: `протокол` + `домен` + `порт`.
* Если пользователь открыл две разные вкладки одного сайта (например, `https://example.com/page1` и `https://example.com/page2`), они делят между собой одно общее хранилище `localStorage`, так как у них совпадает источник.
* Если протоколы различаются (например, `http://example.com` и `https://example.com`), хранилища будут абсолютно изолированы друг от друга.

---

### 2. Сравнение localStorage и sessionStorage

Хотя оба хранилища используют идентичный программный интерфейс (API), они имеют принципиальные различия в области видимости и времени жизни данных.

#### localStorage:
* **Время хранения:** Постоянное (бессрочное). Данные сохраняются после обновления страницы, закрытия вкладки, перезапуска браузера или полной перезагрузки операционной системы. Они могут быть удалены только явно через JavaScript-код или при ручной очистке кеша пользователем.
* **Область действия:** Доступно всем вкладкам, окнам и фреймам, открытым в рамках одного источника (Origin). Изменения, внесенные в одном окне, моментально становятся видимыми в другом.

#### sessionStorage:
* **Время хранения:** Ограничено сессией работы вкладки. Данные сохраняются при перезагрузке страницы, но **полностью удаляются при закрытии вкладки или самого браузера**.
* **Область действия:** Ограничено строго одной текущей вкладкой браузера. Другая вкладка с той же открытой страницей будет иметь свое изолированное хранилище `sessionStorage`.

---

### 3. Программный интерфейс (API) Web Storage
Оба объекта предоставляют одинаковый набор методов и свойств:
* **`setItem(key, value)`** — сохраняет пару "ключ-значение".
* **`getItem(key)`** — считывает значение по указанному ключу. Если ключ отсутствует, возвращает `null`.
* **`removeItem(key)`** — удаляет запись по ключу.
* **`clear()`** — полностью очищает хранилище текущего источника.
* **`key(index)`** — возвращает имя ключа на указанной позиции (полезно для перебора хранилища в цикле).
* **`length`** — возвращает общее количество записанных пар.

---

### 4. Ограничения типов данных и сериализация
Ключевым техническим ограничением `localStorage` и `sessionStorage` является то, что **и ключи, и значения должны быть строго строками**. 

Если попытаться сохранить сложный тип данных (например, массив или объект) напрямую, движок JavaScript автоматически преобразует его в строку методом `.toString()`. В результате объект превратится в бесполезную строку `"[object Object]"`, а исходные данные будут утеряны.

#### Решение проблемы:
Для сохранения сложных структур данных их необходимо предварительно сериализовать (преобразовать в JSON-строку) с помощью метода `JSON.stringify()`, а при чтении — десериализовать обратно с помощью `JSON.parse()`.

---

### 5. Лимиты объема памяти
Большинство современных браузеров выделяют около **5–10 МБ** дискового пространства под данные одного источника. При превышении этого лимита вызов метода `setItem()` выбросит исключение `QuotaExceededError`.

```text
                                |
     +--------------------------+--------------------------+
     |                                                     |
[ localStorage ]                                    [ sessionStorage ]

```

## 3. Практические примеры кода

### Плохой пример: прямое обращение к свойствам объекта и отсутствие сериализации

Обращение к хранилищу как к обычному объекту лишает вас встроенных преимуществ API, а попытка сохранить объект без сериализации приводит к потере структуры данных.

```javascript
const userSession = { name: "Алексей", role: "admin" };

// Плохой подход 1: сохранение объекта напрямую без сериализации
localStorage.setItem("currentUser", userSession); 

// Попытка прочитать данные
const savedUser = localStorage.getItem("currentUser");
console.log(savedUser); // Вывод: "[object Object]" (данные разрушены)

// Плохой подход 2: прямое обращение в обход API
localStorage.theme = "dark"; // Работает, но не позволяет отслеживать ошибки заполнения диска
```

### Хороший пример: создание безопасной обертки с обработкой лимитов памяти

```javascript
// Класс-хелпер для безопасной работы с локальным хранилищем
class StorageHelper {
  // Безопасное сохранение данных с автоматической сериализацией в JSON
  static set(key, value) {
    try {
      const stringifiedValue = JSON.stringify(value);
      localStorage.setItem(key, stringifiedValue);
      return true;
    } catch (error) {
      // Обрабатываем возможное переполнение квоты диска
      if (error.name === "QuotaExceededError") {
        console.error("Ошибка: Браузерное хранилище переполнено!");
      } else {
        console.error("Произошла ошибка при записи данных: ", error.message);
      }
      return false;
    }
  }

  // Безопасное чтение данных с автоматическим разбором JSON
  static get(key) {
    try {
      const value = localStorage.getItem(key);
      // Если ключ не найден, getItem вернет null
      if (value === null) {
        return null;
      }
      return JSON.parse(value);
    } catch (error) {
      console.error("Ошибка разбора JSON при чтении ключа: " + key, error.message);
      return null;
    }
  }

  // Удаление элемента
  static remove(key) {
    localStorage.removeItem(key);
  }
}

// Использование хелпера для сохранения сложного объекта
const settings = {
  theme: "dark",
  fontSize: 16,
  favorites: ["статья 1", "статья 2"]
};

// Запись данных
StorageHelper.set("userSettings", settings);

// Чтение сохраненных данных
const savedSettings = StorageHelper.get("userSettings");

if (savedSettings) {
  console.log(`Настройки успешно загружены. Тема: ${savedSettings.theme}`);
  console.log("Избранные статьи: ", savedSettings.favorites);
}
```

## 4. Вопросы для собеседования

### Вопрос 1: В чем разница между Cookies, `localStorage` и `sessionStorage`?
**Ответ:** 
* **Cookies:** Хранят небольшой объем данных (до 4 КБ), имеют срок жизни (задается сервером), и **автоматически отправляются браузером** на сервер при каждом HTTP-запросе в заголовке `Cookie`, что создает дополнительный сетевой трафик.
* **`localStorage`:** Хранит больший объем данных (5–10 МБ), не имеет срока жизни (бессрочно), данные хранятся локально на клиенте и никогда не отправляются на сервер автоматически.
* **`sessionStorage`:** Работает аналогично `localStorage`, но данные хранятся локально только до тех пор, пока открыта текущая вкладка браузера.

### Вопрос 2: Что произойдет, если попытаться записать объект в `localStorage` без предварительного преобразования в JSON?
**Ответ:** `localStorage` принимает в качестве значений только строки. Если передать обычный объект, движок неявно вызовет метод `.toString()` на этом объекте. В результате в хранилище под вашим ключом запишется стандартная системная строка `"[object Object]"`, а реальные данные (свойства и значения объекта) будут полностью стерты.

### Вопрос 3: Разделяют ли `sessionStorage` две вкладки браузера, в которых открыт один и тот же сайт?
**Ответ:** Нет, не разделяют. `sessionStorage` ограничен контекстом текущей вкладки. Даже если URL, домен и протокол страниц в двух вкладках абсолютно идентичны, у каждой из них будет свое собственное изолированное хранилище `sessionStorage`. Однако, если внутри одной вкладки открыт iframe с того же источника, он будет делить `sessionStorage` с родительской страницей.

### Вопрос 4: Как браузер реагирует на превышение лимита памяти при попытке записать данные в `localStorage`? Как это предотвратить?
**Ответ:** Браузер заблокирует запись и выбросит системное исключение `QuotaExceededError`. Чтобы предотвратить падение приложения, любые операции записи в веб-хранилище (особенно при работе со сложными динамическими данными пользователя) необходимо оборачивать в блок `try...catch` для перехвата этой ошибки и своевременного уведомления пользователя или очистки старого кеша.
