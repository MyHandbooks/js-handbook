JavaScript разрабатывался в спешке, и многие его ранние особенности синтаксиса оказались неудачными, запутанными или небезопасными. Чтобы исправить эти ошибки, не нарушая при этом работу старых сайтов в интернете, в стандарте ECMAScript 5 (2009 год) появился **Строгий режим (Strict Mode)**. Это специальный режим выполнения, который отключает устаревшие особенности языка и превращает скрытые ошибки (silent errors) в явные исключения в консоли.

---

## 1. Глубокое понимание концепции (Core Concept)

### Как включить строгий режим?
Для активации строгого режима используется специальная директива `"use strict";` (или `'use strict';`), написанная в виде обычной строки кода.

#### А. На уровне файла
Директива должна быть написана на **самой первой строчке кода** в файле. Если перед ней будет любой другой исполняемый код (кроме комментариев), строгий режим не включится.

#### Б. На уровне отдельной функции
Директиву можно поместить на первую строчку внутри тела конкретной функции. В этом случае строгий режим будет применяться локально только к коду этой функции.

---

### Где строгий режим включен автоматически?
В современном JavaScript строгий режим часто работает по умолчанию, даже если вы явно не написали директиву `"use strict"`:
1. **Внутри ES-модулей:** Все файлы скриптов, подключенные к HTML как модули (например, `<script type="module">`), всегда выполняются в строгом режиме.
2. **Внутри классов:** Любой код, написанный внутри конструкции `class` (появившейся в ES6), автоматически выполняется в строгом режиме.

---

### Ключевые изменения и ограничения строгого режима

#### 1. Запрет неявного создания глобальных переменных
В обычном режиме, если вы случайно ошиблись в имени переменной при присвоении значения или забыли написать `let`/`const`, JS автоматически создавал глобальную переменную в объекте `window`. В строгом режиме такое действие вызывает явную ошибку `ReferenceError`.

#### 2. Изменение глобального контекста `this`
В обычном режиме, если вызвать простую функцию (не метод объекта), её контекст `this` по умолчанию ссылался на глобальный объект `window`. В строгом режиме контекст <code>this === undefined</code>. Это предотвращает случайное загрязнение глобальной области памяти свойствами.

#### 3. Явные ошибки вместо "тихих" сбоев (Silent Errors)
В обычном режиме попытка перезаписать свойство, доступное только для чтения, или удалить неудаляемые свойства через оператор `delete` просто игнорировалась браузером. В строгом режиме это вызывает явную ошибку `TypeError`.

#### 4. Уникальность параметров функций
В строгом режиме запрещено объявлять функции, у которых параметры имеют одинаковые имена (например, `function sum(a, a, b)`). Это вызовет ошибку `SyntaxError`.

#### 5. Изоляция объекта `arguments` от параметров
В обычном режиме изменение локального параметра функции автоматически меняло соответствующее значение внутри встроенного объекта `arguments`. В строгом режиме эта связь разорвана: объект `arguments` всегда хранит только те исходные значения, которые были переданы в функцию при её вызове.

#### 6. Изоляция области видимости `eval`
Обычно вызов `eval("var x = 10")` создавал переменную `x` во внешней области видимости. В строгом режиме у `eval` появляется собственное изолированное лексическое окружение, и объявленные внутри него переменные не "утекают" наружу.

#### 7. Запрет зарезервированных слов
Запрещено использовать в качестве имен переменных слова, которые зарезервированы под будущие стандарты языка (например: `public`, `private`, `interface`, `protected`, `implements`).

## 2. Ментальная модель

### Сравнение поведения движка в обычном и строгом режимах:

```text
ОБЫЧНЫЙ РЕЖИМ (Sloppy Mode):
[ Код: x = 10; ] -------> [ Движок JS: "Хм, переменной x нет. Создам её в window!" ]
[ Код: delete Object.prototype; ] -------> [ Движок JS: "Нельзя удалить. Промолчу." ] (Тихий сбой)
[ Код: Простая функция show() { this } ] -> [ this указывает на window ]

СТРОГИЙ РЕЖИМ (Strict Mode):
[ Код: x = 10; ] -------> [ Движок JS: "Ошибка! ReferenceError: x is not defined" ] (Громкий сбой)
[ Код: delete Object.prototype; ] -------> [ Движок JS: "Ошибка! TypeError" ] (Громкий сбой)
[ Код: Простая функция show() { this } ] -> [ this указывает на undefined ]
```

## 3. Практические примеры кода

### Плохой пример: опасное поведение обычного режима (Sloppy Mode)

Обычный режим скрывает логические ошибки разработчика, создавая глобальные переменные и молча игнорируя некорректные операции записи в память.

```javascript
// Код работает без "use strict"
function initUser() {
  // Случайная опечатка в имени переменной (пропустили let/const)
  username = "Алексей"; 

  // Попытка удалить неудаляемый объект
  delete Object.prototype; // Ничего не произойдет, ошибки в консоли нет

  console.log("Пользователь инициализирован.");
}

initUser();
// Переменная username утекла в глобальную область видимости
console.log(window.username); // Вывод: "Алексей"
```

### Хороший пример: строгий режим и мгновенная реакция на ошибки

```javascript
"use strict"; // Включаем строгий режим для всего файла

function initUserSecure() {
  try {
    // 1. Попытка присвоить значение необъявленной переменной
    username = "Алексей"; // Вызовет ошибку ReferenceError
  } catch (error) {
    console.error("Поймана ошибка: " + error.message);
  }

  try {
    // 2. Попытка перезаписать свойство, доступное только для чтения
    const admin = {};
    Object.defineProperty(admin, "role", { value: "admin", writable: false });
    admin.role = "user"; // Вызовет ошибку TypeError
  } catch (error) {
    console.error("Поймана ошибка записи: " + error.message);
  }

  // 3. Безопасное поведение this
  function showContext() {
    console.log("Контекст простой функции: ", this); // Выведет: undefined (вместо window)
  }
  showContext();
}

initUserSecure();
```

## 4. Вопросы для собеседования

### Вопрос 1: Что делает директива `"use strict"` и какие основные проблемы она решает?
**Ответ:** Директива `"use strict"` переводит интерпретатор JavaScript в строгий режим работы. Она решает проблемы безопасности и оптимизации кода: запрещает неявное создание глобальных переменных (превращая их в `ReferenceError`), блокирует "тихие" ошибки при попытке записи в свойства только для чтения (превращая их в `TypeError`), делает глобальный контекст `this` равным `undefined` вместо `window` в простых функциях и запрещает использование устаревших конструкций (например, `with`).

### Вопрос 2: В каких случаях строгий режим активируется автоматически без написания директивы?
**Ответ:** Строгий режим активируется автоматически в двух случаях:
1. Внутри всех файлов, которые подключены или работают как **ES-модули** (например, в HTML-теге <code>type="module"</code> или при использовании `import`/`export` в современных сборщиках).
2. Внутри тела любого **класса** (`class`), включая его конструктор, статические и обычные методы.

### Вопрос 3: Как ведет себя контекст `this` при вызове простой функции (не метода объекта) в обычном и строгом режимах?
**Ответ:** 
* В обычном режиме контекст `this` внутри простой функции неявно ссылается на глобальный объект `window` (в браузере) или `global` (в Node.js).
* В строгом режиме это поведение отключено в целях безопасности, и контекст простой функции равен `undefined`.

### Вопрос 4: Как строгий режим влияет на объект `arguments` внутри функций?
**Ответ:** В обычном режиме объект `arguments` и локальные параметры функции связаны: если изменить значение локального параметра (например, <code>a = 10</code>), то значение `arguments[0]` также изменится. В строгом режиме эта связь полностью разорвана — объект `arguments` всегда стабильно хранит исходные значения аргументов, с которыми функция была вызвана, независимо от последующих изменений параметров в теле функции.
