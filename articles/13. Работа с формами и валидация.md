Формы в HTML — это основной инструмент сбора данных от пользователя для последующей отправки на сервер. JavaScript предоставляет богатый программный интерфейс (API) для доступа к элементам форм, динамического контроля фокуса, отслеживания изменений данных и выполнения гибкой валидации перед отправкой.

---

## 1. Глубокое понимание концепции (Core Concept)

### 1. Навигация по элементам формы
Для работы с формами в DOM существует специализированное API, которое работает быстрее и надежнее обычных методов поиска:
* **Коллекция `document.forms`:** Все формы на странице автоматически попадают в эту коллекцию. К форме можно обратиться по ее индексу или по значению атрибута `name`: <code>const registerForm = document.forms.registration</code>.
* **Свойство `form.elements`:** Содержит именованную коллекцию всех элементов управления внутри формы (элементы `<input>`, `<textarea>`, `<select>`, `<button>`, `<fieldset>`). Доступ к конкретному полю осуществляется по его атрибуту `name`: <code>const usernameInput = form.elements.username</code>.

Любое поле ввода хранит обратную ссылку на свою форму в свойстве `input.form`.

---

### 2. Чтение и запись значений
* Для текстовых полей, `<textarea>` и выпадающих списков `<select>` значение считывается и записывается через свойство `value`.
* Для чекбоксов и радиокнопок используется булево свойство `checked`.
* Для `<select>` индекс выбранного пункта доступен в `select.selectedIndex`, а список всех вариантов — в коллекции `select.options`.

---

### 3. События фокусировки: `focus` и `blur`
* **`focus`** срабатывает в момент, когда элемент получает фокус (пользователь кликнул на поле или перешел на него кнопкой Tab).
* **`blur`** срабатывает в момент потери фокуса.

Для ручного управления фокусом используются методы `element.focus()` и `element.blur()`. Текущий сфокусированный элемент на странице всегда доступен в свойстве `document.activeElement`.

#### Важная особенность: всплытие
События `focus` и `blur` **не всплывают**. Если повесить слушатель `focus` на форму `<form>`, он не сработает при фокусе на внутреннем `<input>`. 

Решить эту проблему можно двумя способами:
1. Использовать фазу перехвата (захвата), передав третьим аргументом <code>{ capture: true }</code> при добавлении слушателя.
2. Использовать альтернативные события **`focusin`** и **`focusout`**, которые ведут себя аналогично `focus`/`blur`, но поддерживают стандартное всплытие.

---

### 4. События изменения данных: `input` и `change`
* **`input`:** Срабатывает мгновенно при каждом изменении значения поля (при каждом нажатии клавиши, вставке из буфера или удалении символа). Это лучший выбор для "живой" валидации или поиска на лету.
* **`change`:** Срабатывает только тогда, когда изменение значения зафиксировано. Для текстовых полей это происходит в момент потери фокуса (`blur`). Для чекбоксов, радиокнопок и выпадающих списков `change` срабатывает сразу после выбора нового значения.

---

### 5. Валидация и Constraint Validation API
Для проверки корректности введенных данных современные браузеры предоставляют Constraint Validation API. По умолчанию браузер сам валидирует поля на основе HTML-атрибутов (`required`, `type="email"`, `minlength`, `pattern`) и показывает стандартные всплывающие подсказки.

Чтобы отключить стандартные всплывающие подсказки браузера и реализовать собственную кастомную логику, форме добавляют атрибут `novalidate`.

Ключевые свойства Constraint Validation API:
* **`element.willValidate`** — возвращает `true`, если элемент подлежит валидации при отправке формы.
* **`element.validationMessage`** — содержит локализованный текст ошибки, которую сгенерировал браузер (или которую мы задали вручную). Если поле валидно, возвращается пустая строка.
* **`element.validity`** — объект типа `ValidityState`, содержащий булевы свойства-маркеры ошибок:
  * `valueMissing` — пустое поле при наличии атрибута `required`.
  * `typeMismatch` — введенные данные не соответствуют типу (например, некорректный email).
  * `patternMismatch` — текст не совпадает с регулярным выражением в атрибуте `pattern`.
  * `tooShort` / `tooLong` — длина текста выходит за рамки ограничений `minlength` / `maxlength`.
  * `valid` — возвращает `true`, если элемент прошел все проверки (ошибок нет).

Ключевые методы API:
* **`element.checkValidity()`** — проверяет валидность элемента. Если он не валиден, генерирует на нем событие `invalid` и возвращает `false`.
* **`element.setCustomValidity(message)`** — позволяет задать кастомное текстовое сообщение об ошибке. Если передать непустую строку, поле автоматически помечается как невалидное. Для сброса ошибки и возвращения поля в валидное состояние нужно вызвать метод, передав пустую строку: <code>element.setCustomValidity("")</code>.

---

### 6. Отправка формы: событие `submit`
Событие `submit` генерируется на форме в момент ее отправки (по нажатию на `<button type="submit">` или при нажатии клавиши Enter в текстовом поле).

* **Важно:** Обработчик события `submit` должен всегда вызывать `event.preventDefault()`, чтобы отменить стандартную перезагрузку страницы браузером. После этого данные собираются и отправляются на сервер асинхронно через AJAX/Fetch.
* **Исключение:** Вызов метода <code>form.submit()</code> из JavaScript отправляет форму напрямую на сервер, минуя генерацию события `submit` (обработчики события `submit` на форме при этом не сработают).

## 2. Ментальная модель

Взаимодействие элементов управления формы при обработке и валидации:

```text
[ Форма (HTMLFormElement) ] -- document.forms.registration --+
              |                                              |
              v                                              v
     коллекция elements                              событие 'submit'
              |                                              |
  +-----------+-----------+                                  v
  v                       v                          event.preventDefault()
[ Input: username ]     [ Input: email ]                     |
  - событие 'input'       - событие 'change'                 v
  - свойство validity     - свойство validity      Локальная отправка (fetch)
```

## 3. Практические примеры кода

### Плохой пример: ручная валидация на "keyup" без использования встроенных API

Ручные регулярные выражения для простых полей дублируют работу браузера, а привязка тяжелой логики к событию `keyup` перегружает процессор при быстром вводе.

```javascript
const emailField = document.querySelector("#email-field");
const signupForm = document.querySelector("#signup-form");

// Плохой подход: регулярное выражение вручную и проверка на каждое нажатие клавиши
emailField.addEventListener("keyup", () => {
  const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  // Постоянная манипуляция DOM-стилями на каждый ввод символа
  if (!emailRegex.test(emailField.value)) {
    emailField.style.borderColor = "red"; 
  } else {
    emailField.style.borderColor = "green";
  }
});

signupForm.addEventListener("submit", () => {
  // Забыли вызвать preventDefault, страница перезагрузится, данные будут потеряны
  console.log("Данные отправлены");
});
```

### Хороший пример: использование Constraint Validation API, событий focusin/input и безопасного submit

```javascript
// Получаем форму из коллекции document.forms
const authForm = document.forms.auth;
const emailInput = authForm.elements.userEmail;
const submitBtn = authForm.elements.submitBtn;

// 1. Делегирование фокуса на форме с использованием всплывающего события 'focusin'
authForm.addEventListener("focusin", (event) => {
  // Подсвечиваем активное поле ввода
  event.target.classList.add("focused-field");
});

authForm.addEventListener("focusout", (event) => {
  event.target.classList.remove("focused-field");
});

// 2. Живая валидация с использованием встроенного ValidityState
emailInput.addEventListener("input", () => {
  const feedback = document.querySelector(".email-error-msg");

  // Проверяем встроенные маркеры валидности
  if (emailInput.validity.valueMissing) {
    emailInput.setCustomValidity("Пожалуйста, заполните поле Email.");
  } else if (emailInput.validity.typeMismatch) {
    emailInput.setCustomValidity("Введенный текст не является корректным адресом почты.");
  } else {
    // Обязательно сбрасываем ошибку, чтобы поле снова стало валидным
    emailInput.setCustomValidity(""); 
  }

  // Обновляем текст ошибки в интерфейсе на основе стандартного сообщения
  if (!emailInput.validity.valid) {
    feedback.textContent = emailInput.validationMessage;
    feedback.classList.remove("hidden");
  } else {
    feedback.textContent = "";
    feedback.classList.add("hidden");
  }
});

// 3. Безопасная отправка формы с локальной обработкой
authForm.addEventListener("submit", (event) => {
  // Отменяем стандартное поведение отправки и перезагрузки страницы
  event.preventDefault();

  // Дополнительная финальная проверка всей формы перед AJAX-запросом
  if (!authForm.checkValidity()) {
    console.warn("Форма содержит ошибки. Отправка заблокирована.");
    return;
  }

  // Блокируем кнопку отправки на время запроса
  submitBtn.disabled = true;

  // Собираем данные формы с помощью встроенного объекта FormData
  const formData = new FormData(authForm);
  
  console.log("Отправка данных на сервер через Fetch API...");
  // fetch("/api/login", { method: "POST", body: formData }) ...
});
```

## 4. Вопросы для собеседования

### Вопрос 1: В чем разница между событиями `input` и `change` для текстовых полей?
**Ответ:** 
* Событие `input` срабатывает синхронно в момент любого изменения содержимого поля (ввод символа, удаление, вставка текста).
* Событие `change` для текстовых полей срабатывает отложенно — только тогда, когда пользователь завершил ввод и перевел фокус с этого поля на другой элемент (произошло событие `blur`).

### Вопрос 2: Как обрабатывать события `focus` и `blur` на родительском контейнере, если они не всплывают?
**Ответ:** Для этого существует два решения:
1. Использовать фазу перехвата (capturing), установив флаг `capture: true` третьим аргументом в методе `addEventListener`. Событие пойдет сверху вниз и будет поймано на контейнере-родителе до того, как достигнет поля ввода.
2. Использовать всплывающие аналоги событий — `focusin` и `focusout`. Они работают абсолютно так же, но поддерживают стандартную фазу всплытия (bubbling).

### Вопрос 3: К чему приводит программный вызов метода `form.submit()` из кода?
**Ответ:** Программный вызов метода <code>form.submit()</code> инициирует немедленную отправку данных формы на сервер по адресу, указанному в атрибуте `action`. При этом важно помнить, что браузер отправляет данные напрямую и **не генерирует** событие `submit` на форме. Соответственно, никакие локальные обработчики валидации, привязанные к событию `submit`, выполнены не будут.

### Вопрос 4: Как с помощью Constraint Validation API задать собственное сообщение об ошибке для поля ввода и как его сбросить?
**Ответ:** Кастомное сообщение задается вызовом метода <code>element.setCustomValidity("Текст ошибки")</code>. После этого вызова свойство `element.validity.valid` автоматически принимает значение `false`, а переданный текст записывается в свойство `element.validationMessage`. 

Чтобы сбросить ошибку и вернуть поле в валидный статус, необходимо вызвать этот же метод с пустой строкой: <code>element.setCustomValidity("")</code>.
