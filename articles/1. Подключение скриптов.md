При открытии веб-страницы браузер скачивает HTML-код и начинает читать его сверху вниз, преобразуя текст в объекты (этот процесс называется парсингом HTML), чтобы построить дерево элементов — DOM (Document Object Model). Подключение JavaScript напрямую влияет на этот процесс.

---

## 1. Глубокое понимание концепции (Core Concept)

### Блокировка парсера (Parser Blocking)
Когда браузер натыкается на обычный тег `<script>`, процесс построения DOM полностью останавливается. Браузер не может продолжать сборку страницы, пока JS-файл не скачается из сети и не выполнится движком. Если тяжелый скрипт подключен в начале страницы (внутри тега `<head>`):
1. **Пользователь видит белый экран**, так как HTML-код ниже скрипта еще не обработан и не отрисован.
2. **Возникают ошибки работы с DOM**, если скрипт попытается найти на странице элементы, которые объявлены ниже него по коду (они еще просто не созданы в памяти).

### Историческое решение: Скрипты внизу страницы
Чтобы избежать блокировки, разработчики долгое время помещали обычные скрипты в самый конец документа, прямо перед закрывающим тегом `</body>`. При таком подходе DOM успевает построиться, но скачивание скрипта начинается слишком поздно, что задерживает появление интерактивности на странице.

### Современное решение: Атрибуты `async` и `defer`
Для управления приоритетом загрузки внешних скриптов (у которых есть атрибут `src`) в стандарте HTML5 были введены атрибуты `async` и `defer`.

#### Атрибут `async` (Асинхронный режим)
Браузер скачивает JS-файл в фоновом режиме, не прерывая построение DOM. Однако **сразу после завершения скачивания парсинг HTML приостанавливается**, и скрипт выполняется.
* **Порядок выполнения:** Не гарантирован. Скрипты выполняются по принципу «кто быстрее скачался, тот и выполнился».
* **Применение:** Скрипты, которые не зависят от структуры страницы и друг от друга (метрики, системы аналитики, счетчики, реклама).

#### Атрибут `defer` (Отложенный режим)
Браузер скачивает JS-файл в фоновом режиме. Выполнение скрипта полностью откладывается до момента, пока **HTML-парсер полностью не закончит свою работу** (DOM будет построен).
* **Порядок выполнения:** Скрипты выполняются строго в том порядке, в котором они были объявлены в HTML-документе.
* **Применение:** Основной код приложения, где важен порядок инициализации модулей и гарантированный доступ к готовым DOM-элементам.

#### Модульные скрипты
Скрипты с атрибутом <code>type="module"</code> по умолчанию ведут себя так, как будто у них установлен атрибут `defer` (загружаются асинхронно, выполняются после сборки DOM с сохранением порядка объявления).

---

### Таблица сравнения поведения скриптов

| Характеристика | Обычный `<script>` | `async` | `defer` |
| :--- | :--- | :--- | :--- |
| **Рекомендуемое место размещения** | В конце `<body>` | В `<head>` | В `<head>` |
| **Блокирует ли загрузку DOM?** | **Да** (при загрузке и выполнении) | **Нет** при загрузке, **Да** при выполнении | **Нет** |
| **Порядок выполнения** | По порядку в HTML | По мере скачивания (хаотично) | По порядку в HTML |
| **Доступ к DOM-дереву** | Только если размещен в конце `<body>` | Не гарантирован (зависит от скорости сети) | **Гарантирован** (DOM полностью готов) |
| **Связь с событием `DOMContentLoaded`** | Выполняется до события | Не связан напрямую | Выполняется **строго перед** событием |

## 2. Ментальная модель

Визуализация процесса обработки HTML и JavaScript браузером:

```text
Легенда: 
= - Парсинг HTML (построение DOM-дерева)
* - Фоновое скачивание файла JS по сети
# - Выполнение кода JavaScript движком

Обычный <script src="...">
HTML: ======[Блокировка парсинга]===================>
JS:          **********########

<script async src="..."> (Выполняется по мере готовности)
HTML: ===============[Блокировка]====================>
JS:   **********#####

<script defer src="..."> (Ждет окончания сборки DOM)
HTML: ===============================================>
JS:   **********                      #######
                                      (Событие DOMContentLoaded)
```

## 3. Практические примеры кода

### Плохой пример: блокирующее подключение в `<head>`

Попытка найти кнопку и повесить на нее клик завершится ошибкой, так как на момент работы скрипта кнопка еще не создана в DOM-дереве.

```html
<!-- index.html -->
<!DOCTYPE html>
<html lang="ru">
<head>
  <meta charset="UTF-8">
  <title>Сбой загрузки</title>
  <!-- Обычный скрипт заблокирует парсинг и не найдет кнопку -->
  <script src="app.js"></script>
</head>
<body>
  <button id="btn">Отправить</button>
</body>
</html>
```

```javascript
// app.js
// Скрипт выполняется до парсинга body, btn равен null
const button = document.getElementById("btn");

// Вызовет ошибку: Uncaught TypeError: Cannot read properties of null (reading 'addEventListener')
button.addEventListener("click", () => {
  console.log("Клик зафиксирован");
});
```

### Хороший пример: разделение скриптов на отложенные и асинхронные

```html
<!-- index.html -->
<!DOCTYPE html>
<html lang="ru">
<head>
  <meta charset="UTF-8">
  <title>Правильное подключение</title>
  
  <!-- Основной код приложения загружается в фоне и выполнится после сборки DOM -->
  <script defer src="main.js"></script>
  
  <!-- Аналитика загружается в фоне и выполнится независимо, не блокируя приложение -->
  <script async src="analytics.js"></script>
</head>
<body>
  <button id="btn">Отправить</button>
</body>
</html>
```

```javascript
// main.js
// Благодаря defer кнопка гарантированно существует в DOM на момент выполнения скрипта
const button = document.getElementById("btn");

button.addEventListener("click", () => {
  console.log("Кнопка успешно обработана!");
});
```

## 4. Вопросы для собеседования

### Вопрос 1: В чем разница между `async` и `defer`?
**Ответ:** Оба атрибута позволяют загружать скрипты в фоновом режиме, не блокируя разбор HTML. Разница заключается в моменте их выполнения:
* `async` запускает скрипт сразу же, как только тот скачался. Это приостанавливает парсинг HTML. Порядок выполнения нескольких `async`-скриптов зависит от скорости их скачивания.
* `defer` ждет, пока браузер полностью распарсит HTML-документ, и только тогда запускает скрипты. Они выполняются строго в порядке их объявления в HTML-коде непосредственно перед событием `DOMContentLoaded`.

### Вопрос 2: Что произойдет, если на странице объявлены два скрипта: `<script defer src="A.js">` и `<script defer src="B.js">`, причем файл B скачался быстрее, чем файл A?
**Ответ:** Первым все равно выполнится файл `A.js`. Атрибут `defer` гарантирует сохранение исходного порядка выполнения скриптов, указанного в HTML-документе, независимо от того, какой из файлов физически загрузился по сети быстрее.

### Вопрос 3: Требуется ли явно указывать атрибут `defer` для модульных скриптов?
**Ответ:** Нет. Скрипты с объявленным атрибутом <code>type="module"</code> по умолчанию обладают поведением `defer` (загружаются асинхронно в фоне и выполняются только после завершения построения DOM-дерева в порядке их объявления).
