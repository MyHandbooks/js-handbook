Замыкание — одна из фундаментальных концепций JavaScript, определяющая, как функции работают с оперативной памятью и областями видимости. Понимание замыканий требует детального разбора скрытых механизмов работы движка JS.

---

## 1. Глубокое понимание концепции (Core Concept)

### Что такое замыкание?
**Замыкание (Closure)** — это способность функции запоминать свою внешнюю область видимости (переменные родительского контекста) и иметь к ним доступ даже тогда, когда она выполняется вне этой родительской области.

В JavaScript все функции по умолчанию являются замыканиями. Они автоматически «запоминают» место своего создания.

---

### Механизм работы: Скрытое свойство `[[Environment]]`

Жизненный цикл функции и её связь с памятью состоят из трех основных фаз:

#### А. Фаза создания функции (Function Creation)
Когда функция объявляется в коде, движок создает её объект в куче. В этот момент функция автоматически получает скрытое служебное свойство — **`[[Environment]]`**. 
* Движок JS записывает в `[[Environment]]` прямую ссылку на то **Лексическое окружение (Lexical Environment)**, в котором эта функция была физически написана.

#### Б. Фаза вызова функции (Function Execution)
Каждый раз, когда функция запускается, для неё создается новое локальное лексическое окружение (Lexical Environment), состоящее из записи локальных переменных (`Environment Record`) и ссылки на внешнее окружение (`Outer Reference`).
* **Важно:** Ссылка на внешнее окружение `Outer Reference` для этого нового запуска автоматически берется напрямую из скрытого свойства `[[Environment]]` вызываемой функции.

#### В. Фаза сборки мусора и сохранения памяти
Обычно после того, как функция завершает работу, её локальное лексическое окружение удаляется из памяти автоматическим сборщиком мусора (Garbage Collector), так как на него больше никто не ссылается.
* **Исключение:** Если родительская функция создала и вернула наружу дочернюю функцию, лексическое окружение родителя **не может быть удалено**. Дочерняя функция (записанная во внешнюю переменную) через своё скрытое свойство `[[Environment]]` удерживает прямую ссылку на окружение родителя. Оно остается достижимым для сборщика мусора и продолжает жить в памяти.

---

### Ловушка на собеседованиях: Замыкание в цикле с `var`

Классический пример, демонстрирующий разницу областей видимости:

```javascript
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100); 
}
// Выведет в консоль: 3, 3, 3 (а не 0, 1, 2)
```
Поскольку `var` имеет функциональную область видимости, переменная `i` создается в единственном экземпляре на всю функцию (или глобальную область). `setTimeout` откладывает выполнение колбэков на 100 мс. Цикл завершается мгновенно, и общее значение `i` в памяти становится равным `3`. Когда колбэки запускаются, они все обращаются к одной и той же переменной `i` со значением `3`.

Если заменить `var` на `let`, то при каждой итерации цикла будет создаваться **новое лексическое окружение со своей независимой копией переменной `i`**, на которую и закроется соответствующий колбэк.

## 2. Ментальная модель

### Схема связей лексических окружений в оперативной памяти:

Когда мы создаем счетчик <code>const counter1 = createCounter()</code> и вызываем его:

```text
+-------------------------------------------------------------+
| ГЛОБАЛЬНОЕ ОКРУЖЕНИЕ (Global Lexical Environment)           |
|  - Переменная counter1 = [ссылка на дочернюю функцию]       |
+-------------------------------------------------------------+
                               ^
                               | (Связь Outer Link через свойство [[Environment]])
+-------------------------------------------------------------+
| ОКРУЖЕНИЕ ВЫЗОВА createCounter() (Родительское)             |
|  - Переменная count = 2                                     |
+-------------------------------------------------------------+
                               ^
                               | (Связь Outer Link)
+-------------------------------------------------------------+
| ЛОКАЛЬНОЕ ОКРУЖЕНИЕ ВЫЗОВА counter1() (Дочернее)            |
|  - Локальных переменных нет                                 |
+-------------------------------------------------------------+
```

## 3. Практические примеры кода

### Плохой пример: классическая утечка контекста в цикле из-за `var`

Использование `var` ломает логику асинхронного выполнения, заставляя все функции ссылаться на одно и то же изменяемое значение в памяти.

```javascript
function runInsecureLoop() {
  for (var i = 0; i < 3; i++) {
    // Каждая анонимная функция замыкается на единственную переменную i
    setTimeout(function() {
      // К моменту запуска i уже равно 3
      console.log("var-индекс:", i); 
    }, 100);
  }
}
runInsecureLoop(); // Выведет: var-индекс: 3 (три раза)
```

### Хороший пример: генератор независимых счетчиков и безопасный цикл

```javascript
// 1. Создание изолированного состояния с помощью замыкания
function createCounter() {
  // Переменная count скрыта от внешнего мира внутри этого лексического окружения
  let count = 0; 
  
  return function() {
    count++; // Функция имеет доступ к переменной родителя
    return count;
  };
}

// Создаем первый независимый счетчик
const counter1 = createCounter(); 
console.log(counter1()); // Выведет: 1
console.log(counter1()); // Выведет: 2

// Создаем второй счетчик. Он получает свое собственное лексическое окружение в памяти
const counter2 = createCounter();
console.log(counter2()); // Выведет: 1 (абсолютно независимый результат)


// 2. Безопасный асинхронный цикл с использованием let
function runSecureLoop() {
  for (let i = 0; i < 3; i++) {
    // При каждой итерации создается отдельное лексическое окружение с собственным значением i
    setTimeout(function() {
      console.log("let-индекс:", i); // Выведет корректно: 0, 1, 2
    }, 100);
  }
}
runSecureLoop();
```

## 4. Вопросы для собеседования

### Вопрос 1: Что такое замыкание в JavaScript?
**Ответ:** Замыкание — это функция, которая запоминает переменные из своей внешней области видимости (где она была физически объявлена) и имеет к ним доступ даже тогда, когда она вызывается вне этой области видимости. Под капотом это реализуется за счет скрытого свойства `[[Environment]]` объекта функции, которое хранит ссылку на родительское лексическое окружение.

### Вопрос 2: Будут ли два счетчика, созданные из одной функции-генератора, делить одну и ту же переменную в памяти?
**Ответ:** Нет, не будут. Каждый раз, когда вызывается родительская функция (например, `createCounter()`), в памяти создается абсолютно новое, изолированное лексическое окружение (`Lexical Environment`) со своим собственным набором локальных переменных. Возвращаемая дочерняя функция замыкается именно на это новое окружение, поэтому счетчики работают полностью независимо друг от друга.

### Вопрос 3: Как замыкания могут приводить к утечкам памяти (Memory Leaks)?
**Ответ:** Дочерняя функция удерживает ссылку на лексическое окружение родительской функции через свойство `[[Environment]]`. До тех пор, пока в программе существует ссылка на дочернюю функцию (например, она записана в глобальную переменную), лексическое окружение родителя со всеми его переменными не может быть удалено сборщиком мусора. Если родительская функция хранила внутри себя тяжелые данные (например, большие массивы или DOM-элементы), они будут оставаться в памяти, вызывая утечку.

### Вопрос 4: Как работает сборщик мусора (Garbage Collector) с лексическими окружениями завершивших работу функций?
**Ответ:** Сборщик мусора работает по принципу досягаемости (Reachability). Обычно после завершения функции её локальное окружение становится недосягаемым из глобального контекста и удаляется. Однако, если из функции возвращается другая функция (или объект), которая ссылается на это окружение, оно остается досягаемым через цепочку ссылок. Сборщик мусора не удаляет такое окружение из памяти до тех пор, пока жива ссылка на возвращенный объект.
