События — это сигналы от браузера, указывающие на то, что на веб-странице что-то произошло (например, пользователь кликнул по кнопке, ввел текст, прокрутил страницу или документ полностью загрузился). JavaScript позволяет перехватывать эти сигналы и реагировать на них с помощью функций-обработчиков (event listeners).

---

## 1. Глубокое понимание концепции (Core Concept)

### 1. Способы назначения обработчиков событий

Существует три способа связать событие с функцией в JavaScript:

1. **Атрибут HTML:** Код обработчика пишется прямо внутри HTML-тега: 
   <code>&lt;button onclick="alert('Клик!')"&gt;Клик&lt;/button&gt;</code>. 
   *Минусы:* Смешивание разметки и логики, неудобно поддерживать, нельзя повесить несколько функций на одно событие.
2. **DOM-свойство:** Назначение функции через свойство DOM-элемента в JS: 
   <code>button.onclick = function() { ... }</code>. 
   *Минусы:* Ограничение в один обработчик. Каждое новое присвоение <code>button.onclick = newFunc</code> полностью заменяет предыдущую функцию.
3. **Метод `addEventListener`:** Современный стандарт: 
   <code>button.addEventListener('click', handler)</code>.
   *Плюсы:* Позволяет назначать неограниченное количество обработчиков на одно событие, поддерживает управление фазами события и корректно удаляет слушатели через `removeEventListener`.

---

### 2. Объект события `Event`
Когда событие происходит, браузер автоматически создает объект события `Event`, записывает в него метаданные и передает первым аргументом в функцию-обработчик.

Ключевые свойства объекта `Event`:
* `type` — тип события (например, `"click"`, `"submit"`).
* `target` — исходный целевой элемент, на котором *возникло* событие (самый глубокий тег, по которому кликнули).
* `currentTarget` — элемент, на котором в данный момент *сработал* обработчик (тег, на который была повешена функция через `addEventListener`).
* `bubbles` — булево значение, указывающее, всплывает ли это событие вверх по DOM-дереву.
* `cancelable` — показывает, можно ли отменить поведение события по умолчанию.
* `defaultPrevented` — возвращает `true`, если стандартное поведение события было отменено.

---

### 3. Распространение события (Event Flow / Propagation)

Когда вы кликаете по кнопке, находящейся внутри нескольких контейнеров, событие проходит три последовательные фазы:

1. **Фаза погружения (Capturing phase):** Событие спускается от самого верха (`window`, `document`) вниз по дереву к целевому элементу. По умолчанию обработчики не реагируют на этой фазе. Чтобы перехватить событие на фазе погружения, в `addEventListener` третьим параметром передают флаг <code>{ capture: true }</code>.
2. **Фаза цели (Target phase):** Событие достигает целевого элемента (того, на котором кликнули). На этой фазе срабатывают обработчики, установленные на самом элементе.
3. **Фаза всплытия (Bubbling phase):** Событие начинает подниматься обратно от целевого элемента к самому верху (`window`). Большинство обработчиков в реальной разработке работают именно на этой фазе.

---

### 4. Управление ходом события

* **`preventDefault()`** — отменяет стандартное поведение браузера для этого события (например, переход по ссылке `<a>` или отправку формы `<form>` на сервер). Само событие при этом продолжает всплывать.
* **`stopPropagation()`** — останавливает дальнейшее всплытие (или погружение) события. Родительские элементы больше не узнают о том, что событие произошло. Однако, если на текущем элементе висит несколько обработчиков на это же событие, все они успешно выполнятся.
* **`stopImmediatePropagation()`** — полностью останавливает обработку события: не только предотвращает его всплытие наверх, но и мгновенно блокирует выполнение всех остальных обработчиков этого события на текущем элементе.

---

### 5. Делегирование событий (Event Delegation)

Делегирование — это прием оптимизации производительности, основанный на механизме всплытия событий. 

Вместо того чтобы вешать индивидуальный обработчик на каждую из 100 кнопок в списке, мы вешаем всего один обработчик на их общего родителя. Когда пользователь кликает на кнопку, событие всплывает до родителя, где мы перехватываем его, анализируем <code>event.target</code> и, если это кнопка, выполняем нужную логику.

---

### 6. Пользовательские события (CustomEvent)

JavaScript позволяет создавать и генерировать собственные события с помощью конструктора `CustomEvent`. Это полезно для создания независимых компонентов (модулей), которые общаются друг с другом через события.
* Для передачи полезной нагрузки (данных) используется специальное свойство `detail` во втором аргументе конструктора.
* Триггер события осуществляется методом `dispatchEvent()`.

## 2. Ментальная модель

Визуализация прохода события через фазы:

```text
           Фаза Погружения (1. Capturing)
          [ window ]  -------------------------+
              |                                |
              v                                |
         [ document ]                          |
              |                                |
              v                                |
          [ body ]                             v
              |                         Фаза Всплытия (3. Bubbling)
              v                                |
           [ div ]                             |
              |                                |
              +======> [button] <==============+
                 Фаза Цели (2. Target Phase)
```

## 3. Практические примеры кода

### Плохой пример: регистрация обработчиков в цикле и ручное переопределение свойств

```javascript
// Находим 100 кнопок на странице
const buttons = document.querySelectorAll(".btn-item");

// Плохой подход: вешаем 100 отдельных функций-обработчиков в память
buttons.forEach((btn) => {
  // Использование свойства onclick затирает предыдущие обработчики
  btn.onclick = function(e) {
    console.log("Клик по кнопке: " + e.target.textContent);
  };
});

// Отмена перехода по ссылке старым ненадлежащим образом
const link = document.querySelector(".my-link");
if (link) {
  link.onclick = function() {
    // Возврат false вместо preventDefault не всегда надежен и не поддерживается современными API
    return false; 
  };
}
```

### Хороший пример: делегирование событий, безопасная отмена и кастомные события

```javascript
// 1. Делегирование событий на общем родителе
const listContainer = document.querySelector(".interactive-list");

if (listContainer) {
  listContainer.addEventListener("click", (event) => {
    // Находим ближайшую кнопку, даже если кликнули по иконке внутри нее
    const button = event.target.closest(".btn-action");

    // Если кликнули мимо целевых элементов, игнорируем событие
    if (!button || !listContainer.contains(button)) return;

    console.log("Обработана кнопка с ID: " + button.dataset.id);
  });
}

// 2. Безопасная отмена стандартного поведения и остановка всплытия
const form = document.querySelector(".signup-form");

if (form) {
  form.addEventListener("submit", (event) => {
    // Отменяем стандартную перезагрузку страницы браузером
    event.preventDefault(); 
    
    // Останавливаем событие, чтобы родительские контейнеры не реагировали на отправку
    event.stopPropagation(); 
    
    console.log("Форма отправлена локально через AJAX.");
  });
}

// 3. Создание и генерация пользовательского события (CustomEvent)
const chatWidget = document.querySelector(".chat-widget");

if (chatWidget) {
  // Создаем событие с указанием всплытия и передачей данных в detail
  const messageEvent = new CustomEvent("newMessage", {
    detail: { text: "Привет всем!", author: "Алексей" },
    bubbles: true,
    cancelable: true
  });

  // Инициируем вызов события на элементе
  chatWidget.dispatchEvent(messageEvent);
}

// Слушаем пользовательское событие на уровне документа (благодаря bubbles: true)
document.addEventListener("newMessage", (event) => {
  // Безопасно извлекаем данные из свойства detail
  console.log(`Новое сообщение от ${event.detail.author}: ${event.detail.text}`);
});
```

## 4. Вопросы для собеседования

### Вопрос 1: В чем разница между свойствами `event.target` и `event.currentTarget`?
**Ответ:** 
* `event.target` — это целевой элемент, который непосредственно инициировал событие (самый глубокий тег, на который пришелся физический клик пользователя). Он не меняется в процессе всплытия.
* `event.currentTarget` — это элемент, на котором в данный момент выполняется обработчик события (элемент, на который был повешен метод `addEventListener`). По мере всплытия события это свойство меняется, указывая на каждого родителя по очереди.

### Вопрос 2: Как работают методы `stopPropagation()` и `stopImmediatePropagation()`, и в чем их разница?
**Ответ:** 
* `stopPropagation()` останавливает распространение события дальше по дереву DOM (вверх на фазе всплытия или вниз на фазе погружения). Событие "застревает" на текущем элементе.
* `stopImmediatePropagation()` делает то же самое, но дополнительно мгновенно останавливает выполнение всех остальных обработчиков этого же события, которые установлены на *этом же* элементе ниже по коду.

### Вопрос 3: В чем суть приема "делегирование событий" и какие преимущества он дает?
**Ответ:** Делегирование событий — это подход, при котором один обработчик вешается на родительский элемент для контроля событий на его дочерних элементах (благодаря фазе всплытия). 

Преимущества:
1. **Экономия оперативной памяти:** Вместо сотен индивидуальных функций в памяти хранится всего одна.
2. **Динамичность:** Не нужно заново вешать обработчики событий на новые дочерние элементы, добавленные в DOM через JavaScript (они начинают работать автоматически).
3. **Упрощение кода:** Меньше вызовов `addEventListener` и `removeEventListener`.

### Вопрос 4: Как передать кастомные данные при генерации собственного события?
**Ответ:** Для генерации кастомных событий используется класс `CustomEvent`. Данные передаются во втором аргументе-конфигураторе в виде свойства `detail` (это ограничение стандарта для защиты от хаотичной структуры данных):
<code>const ev = new CustomEvent('test', { detail: { info: 'данные' } })</code>. На принимающей стороне данные считываются через <code>event.detail.info</code>.
